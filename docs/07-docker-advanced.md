# 07. Docker 完全実践マニュアル：基礎理論から実践ハンズオンまで

本マニュアルは、Dockerの「基礎概念と運用ルール」という理論と、「実際にゼロから手を動かして環境を作る」という実践ハンズオンを融合させた**完全版のガイド**です。
「なんとなくコンテナを使っている」状態から脱却し、環境構築を5倍効率化する本質的なスキルを身につけます。

---

## 第1章: Dockerの基本概念と導入メリット

なぜDockerを導入すると開発や研究が圧倒的に効率化するのでしょうか？
それは、「自分のPCでは動くが他人の環境では動かない」という依存関係のトラブルをゼロにし、誰が立ち上げても全く同じ環境が数分で構築できるからです。

Dockerを理解する上で、以下の2つの違いを明確に区別してください。

* **イメージ（設計図・ゲームソフト）**
  アプリケーションを動かすために必要なすべて（ソースコード、OS、ライブラリ、各種設定）が含まれた静的なパッケージです。一度作られた層はキャッシュとして保存・再利用されます。
* **コンテナ（実行環境・ゲーム機）**
  イメージを元にして、実際に起動・作動している「動く仮想環境」の箱です。この箱は外界から独立（隔離）しており、Mac本体を汚しません。1つのイメージから独立した複数のコンテナを起動することも可能です。

## 第2章: はじめてのコンテナ操作と体験（ハンズオン）

まずはネット上に存在する既存のイメージを取得し、コンテナとして動かしてみましょう。

### 1. hello-world を動かす
ターミナルで以下を実行します。
```bash
docker pull hello-world
docker run hello-world
```
一瞬で `Hello from Docker!` と表示されます。これは、`docker pull` でネットからイメージ（設計図）を取得し、`docker run` で隔離されたコンテナとして起動した結果です。

### 2. コンテナの起動オプションとポートマッピング
次に、少し複雑な「データベース」を立ち上げてみます。

```bash
docker pull postgres
docker run --name my-postgres -e POSTGRES_PASSWORD=mysecretpassword -p 9432:5432 -d postgres
```
ここでは重要なオプションを付与しています。
* `--name`: コンテナに `my-postgres` というわかりやすい名前をつけます。
* `-e`: コンテナ内に環境変数を渡します（この場合はDBの初期パスワード）。
* `-d`: **バックグラウンド（デタッチ）モード**。ターミナルを占有せずに裏側で常時稼働させます。
* `-p 9432:5432`: **ポートマッピング**。Mac側の `9432` 番ポートへのアクセスを、コンテナ内の `5432` 番へ転送します。

> 💡 **【重要】ポートの選び方のベストプラクティス**
> ポートは自由ですが、トラブルを防ぐため以下のルールを守るのがプロの基本です。
> - ❌ `< 1024` : システム用の予約ポートなので使わない。
> - ❌ `3306` `5432` など : 競合の危険があるため、Mac側（左側）にはそのまま指定しない。
> - ✅ `8000番台` : WebサーバーやAPI用（8000, 8080等）に最適。
> - ✅ `9000番台` : データベースやキャッシュ用（9432, 9306等）へズラして設定すると、競合を100%回避できる。

### 3. 実践体験：最強のデータサイエンス環境（Jupyter）を立ち上げる
部分的なデータベースだけでなく、R言語やJulia、Pythonの重い統計ライブラリが「全部入り」になった巨大な分析環境も、コマンド一つで手に入ります。数学科や情報系の学生にとっては、この環境が最も身近に活躍するはずです。

ターミナルで以下のコマンドを実行します。
```bash
docker run -p 8888:8888 -v "$PWD":/home/jovyan/work jupyter/datascience-notebook
```

ここでは `-v "$PWD":/home/jovyan/work` という新しいオプション（ボリュームマウント）が登場しています。
* **`-v ホストのディレクトリ:コンテナのディレクトリ`**
  Mac側の「現在のフォルダ(`$PWD`)」と、仮想環境の「作業フォルダ」を**直結・同期**します。
  これにより、ブラウザ上のJupyterで書いたノートブックや分析データが、コンテナを終了・削除してもMacの手元にしっかりとファイルとして残るようになります。

1. コマンド実行後、数GBの環境一式がダウンロードされて起動します。
2. ターミナルに表示された `http://127.0.0.1:8888/lab?token=...` というURLをコピーし、ブラウザで開きます。
3. Python、R、Juliaが全てセットアップされたJupyterLabが即座に使えます。
4. 作業が終わってターミナルで `Ctrl + C` を押せば環境は消滅し、Macは完全にクリーンな状態のまま保たれます。

### 4. コンテナの「中」に入る体験
起動中のコンテナは、ひとつの一時的なPCのようなものです。先ほど裏側（`-d`）で立ち上げた `my-postgres` コンテナの中に入ってみましょう。
```bash
docker exec -it my-postgres bash
```
プロンプトが `root@xxxx...:/#` に変われば成功です。ここは完全に隔離されたLinuxの仮想空間です。確認ができたら `exit` と打ってMacの世界に戻ります。

### 5. 日常的な管理と終了操作
```bash
# 現在「起動中」のコンテナを確認
docker ps

# 停止中のものを含めた全コンテナを確認
docker ps -a

# コンテナを安全に停止する（推奨）
docker stop my-postgres

# 不要なイメージをまとめて削除（ディスク容量の掃除）
docker image prune
```
※ `docker kill` という強制終了コマンドもありますが、データ破損のリスクがあるため、どうしても停止しないような緊急時以外は使用しません。

## 第3章: 自分専用のイメージを作る（Dockerfile 完全解剖）

次は、自作のPythonアプリやNode.jsアプリを組み込んだオリジナルのイメージを作ります。そのための設計書が **`Dockerfile`** （拡張子なし）です。

### 1. 効率的なディレクトリ構造の標準化
ごちゃごちゃにならないよう、実務を見据えた構造を意識します。

```text
my-project/
├── Dockerfile             # 設計図
├── requirements.txt       # （Node.jsなら package.json など依存関係ファイル）
├── src/
│   └── main.py            # ソースコード
├── .dockerignore          # 除外ファイル設定
└── docker-compose.yml     # 後述（複数コンテナ用）
```

### 2. Dockerfile の主要命令と「キャッシュ活用の黄金律」
`Dockerfile` は上から順番に実行されます。ここには**ビルド速度を5倍にするための決定的なコツ**があります。
以下はPythonの例ですが、Node.js等でも全く同じ理屈です。

```dockerfile
# ① ベースイメージの指定（必ず一番上に書く）
FROM python:3.12-alpine

# ② コンテナ内の作業ディレクトリを用意
WORKDIR /app

# ③ 【超重要】パッケージ一覧ファイルだけを「先に」コピーする
COPY requirements.txt .

# ④ 必要なライブラリをインストールする（重い処理）
RUN pip install -r requirements.txt

# ⑤ アプリケーションのコード全体を「最後に」コピーする
COPY src/ ./src/

# ⑥ このアプリが待ち受けるポートの明示（ドキュメント的役割）
EXPOSE 8000

# ⑦ コンテナ起動時のデフォルトコマンド
CMD ["python", "src/main.py"]
```

#### なぜ「依存ファイル」だけを先にコピーするのか？
Dockerは一度実行した行をキャッシュし、次回ビルドを高速化します。
しかし、ファイルの変更を検知すると「その行以降のキャッシュはすべて無効」になります。

開発中、最も頻繁に書き換えるのは「ソースコード（`main.py` など）」です。
もし全てをまとめて最初に COPY してしまうと、コードを1文字直しただけで毎回重たい `pip install` がやり直しになります。
しかし、上記のように `requirements.txt` と `install` を上部に隔離しておけば、ソースコードをいじっただけの再ビルドなら**インストール工程が丸ごとスキップされ、1秒でビルドが完了**するようになります。

### 3. `.dockerignore` は必須
Dockerfileと同じ場所に作り、コンテナに持っていきたくないものを書きます。これをしないとビルドが遅くなり、イメージサイズが肥大化します。
```text
node_modules/
.venv/
.git/
.env
```

### 4. ビルドと実行
```bash
# 設計図からイメージを作成 ("my-api" という名前をつける)
# 最後の「.」は現在のディレクトリを指します。
docker build -t my-api .

# 起動
docker run -d -p 8000:8000 my-api
```

## 第4章: 複数サービス統合と Compose 完全解剖

現実の開発では「Webアプリを立ち上げ」「データベースを立ち上げ」「キャッシュサーバーを立ち上げ」…と複数のコンテナが必要です。
これらを毎回 `docker run` コマンドで起動するのは不可能に近いため、YAMLファイル（`docker-compose.yml`）で全てを一元管理する **Docker Compose** が登場します。

### 1. 黄金のテンプレート（Webアプリ ＋ DB）

プロジェクトのルートに `docker-compose.yml` を作成し、以下のように記述します（YAMLはインデントがスペース2個単位であることに注意してください）。

```yaml
version: '3.8'

services: # 各コンテナ（サービス）の設定

  # ① 自分たちで作るWebアプリ
  web:
    build: .             # Dockerfileをもとにビルドする
    ports:
      - "8080:8000"      # Macの8080をコンテナの8000へ繋ぐ
    volumes:
      - ./src:/app/src   # バインドマウント（Mac側の編集をコンテナ内に即座に反映・ホットリロードさせる）
    environment:
      - DATABASE_URL=postgresql://user:password@db:5432/app_db
    depends_on:          # dbコンテナが立ち上がってからwebを起動
      - db

  # ② データベース
  db:
    image: postgres:15-alpine # 既存の公式イメージをそのまま使う
    ports:
      - "9432:5432"           # DBのポートは9000番台にズラしてMac側と競合しないようにする
    environment:
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=password
      - POSTGRES_DB=app_db
    volumes:
      - db_data:/var/lib/postgresql/data # 名前付きボリューム（コンテナが消えてもデータを永続化）

# 永続化ストレージの宣言（DB用）
volumes:
  db_data:
```

### 2. コンテナ間ネットワーク連動の魔法
Docker Compose最大の魅力は、**「独自の内部ネットワークの自動生成」**です。
上の `environment` をよく見てください。
`postgresql://user:password@db:5432/app_db`
接続先のホスト名として、普通ならIPアドレスを書くべき場所に **`db`** と記述しています。

Docker Composeのネットワーク内では、**定義したサービス名がそのままDNSホスト名として名前解決**されます。さらに、内部同士の通信は `-p` によるポート開放がなくても自由に疎通できます。

### 3. 実践運用コマンド
ディレクトリ内で以下のコマンドを実行するだけで、オーケストラの指揮者のように全サービスが統括されます。

* **全てのコンテナを裏側で一斉起動する**
  ```bash
  docker compose up -d
  ```
* **状態確認**
  ```bash
  docker compose ps
  ```
* **【重要】強制再ビルドして起動する**
  ```bash
  docker compose up -d --build
  ```
  ソースコードに新しいパッケージを追加したり、Dockerfileを書き換えた際は、単に `up` を打つだけでは古いキャッシュが使われてしまいます。変更があれば必ず `--build` をつけて一式を更新します。
* **全てを停止し、綺麗にお片付けする**
  ```bash
  docker compose down
  ```
  ※ 定義した `volumes` (データ) は保持されたまま、コンテナとネットワークのみが削除され安全に初期化されます。

# 第1章 はじめに：LaTeXとは？

### なぜ、我々はWordを使わないのか？

Microsoft Wordは、一般的なビジネス文書やチラシを作るには優れたソフトです。しかし、我々が目指す「数理的な文書作成」においては、致命的な弱点を抱えています。

「数式が画像のように扱われ、行間がガタガタになる」
「100ページを超えると動作が重くなり、図表番号がズレる」
「バージョンが変わるとレイアウトが崩れる」

これらは、Wordが「見た目（WYSIWYG: What You See Is What You Get）」を優先して設計されているがゆえの限界です。
対して、LaTeXは**「論理構造」を優先して設計された、学術文書のためのシステム**です。

### Word vs LaTeX 決定的な違い

| 特徴 | Microsoft Word | LaTeX |
| --- | --- | --- |
| **設計思想** | **WYSIWYG**（見たままが得られる） | **WYSIWYM**（意味した通りのものが得られる） |
| **得意分野** | 短い文書、チラシ、ビジネスレター | **論文、書籍、スライド、複雑な数式** |
| **数式の美しさ** | 普通（調整が難しい） | **極めて美しい（世界標準の組版品質）** |
| **長文耐性** | ページが増えると動作が重くなる | **数百・数千ページでも極めて安定** |
| **互換性** | バージョン間でレイアウト崩れのリスク | **30年前のファイルも最新環境で動く** |
| **コスト** | 有料（Officeライセンスが必要） | **無料（オープンソース）** |

### LaTeXを学ぶべき「真の理由」

単に「レポートの指定だから」という理由だけで使うのはもったいないことです。LaTeXには、数理系学生を虜にする圧倒的なメリットがあります。

### 1. 圧倒的に美しい数式

LaTeXで出力された数式は、プロフェッショナルな品質を持っています。積分記号の傾き、変数のイタリック体、添字の配置……すべてが計算されたバランスで配置されます。これは世界中の数学者・物理学者が何十年もかけて磨き上げてきた組版技術の結晶です。

### 2. 「中身」に集中できる

「見出しのフォントサイズはどうするか？」「余白は何ミリか？」といったデザインの悩みから解放されます。あなたは**「論理構造（何を書くか）」に全集中**し、「どう見せるか（デザイン）」はLaTeXに任せることができます。

### 3. プレーンテキストの堅牢性

LaTeXのファイル（`.tex`）はただのテキストデータです。専用ソフトがなくても開くことができ、Gitなどのバージョン管理システムとも相性が抜群です。「ファイルが壊れて開けない」という悲劇とは無縁です。

> 📝 補足：クラウドツールという選択肢
OverleafやCloud LaTeXなどのクラウドサービスを使えば、環境構築なしでブラウザ上でLaTeXを使えます。共同執筆にも便利です。ただし、本ガイドではローカル環境の構築を扱います。ローカル環境には「オフラインで作業できる」「Gitで細かくバージョン管理できる」「自分好みにカスタマイズできる」といった利点があります。
> 

### LaTeXの仕組み：執筆＝プログラミング

LaTeXでの文書作成は、プログラミングそのものです。

1. **ソースコードを書く（Source Code）**
    - エディタ（VS Codeなど）で、文章とコマンドを記述します（拡張子は `.tex`）。
2. **コンパイルする（Compile）**
    - **組版エンジン**がソースコードを読み込み、数式や文字の配置を厳密に計算します。
3. **PDF生成（Output）**
    - 誰が見ても、どのプリンタで印刷しても1ミリのズレもない、完璧なPDFが出力されます。

# 第2章 環境構築：TeX Live (MacTeX) の導入と管理

数理系の学生にとって、この第2章が最初の難関であり、同時に**「研究者としての第一歩」**でもあります。

まずは用語を整理しましょう。初心者が混乱しやすい「3つのキーワード」があります。

### 1. 基礎知識：3つの重要キーワード

### ① TeX Live（テフ ライブ）

**「本体」**です。
世界中で標準的に使われているLaTeXのディストリビューション（配布パッケージ）。コンパイルに必要なエンジン、数千種類のパッケージ、フォントなどがすべて詰まっています。Windows, Linux, Macですべて共通の環境を提供します。

### ② MacTeX（マックテフ）

**「Mac専用の詰め合わせパック」**です。
上記の「TeX Live」を、Macで簡単にインストール・利用できるようにパッケージングしたものです。Macユーザーは、これを入れることで裏側にある「TeX Live」環境を手に入れることになります。

### ③ Homebrew（ホームブリュー）

**「Macのパッケージ管理ツール」**です。
MacTeXのような巨大なソフトや、プログラミング言語（Python, C++など）を、ターミナルのコマンド一つでインストール・管理できるツールです。「エンジニアのためのApp Store」のような存在です。

---

### 2. インストール手順

本ガイドでは、将来の研究活動を見据えて、**Homebrewを経由してMacTeXをインストールする方法**を採用します。これにより、環境の更新や削除が容易になります。

### Step 1: Homebrewの導入・確認

まず、あなたのMacに「管理ツール」が入っているか確認します。

1. **ターミナルを開く**
    - `[Cmd]` + `[Space]` を押し、「ターミナル」と入力してEnter。
2. **Homebrewの確認**
    - 以下のコマンドを入力してEnterを押します。
    
    ```bash
    brew --version
    ```
    
    - `Homebrew 4.x.x` のように表示されればOKです。**Step 2へ進んでください。**
    - `command not found` と出た場合は、以下の手順でHomebrewをインストールします。
3. **Homebrewのインストール（未導入の場合のみ）**
    1. [Homebrew公式サイト](https://brew.sh/ja/)にアクセス
    2. ページ中央にある「インストール」の下の長いコマンドをコピー
    3. ターミナルに貼り付けて `Enter` を押す
    4. パスワードを求められたら、Macのログインパスワードを入力（画面には表示されませんが入力されています）
    5. 途中で `Press RETURN to continue` と表示されたら `Enter` を押す
    6. インストール完了後、画面に表示される `Next steps` の指示に従い、パスを通すコマンドを実行
    7. ターミナルを一度閉じて開き直し、`brew --version` で導入を確認

### Step 2: MacTeX (TeX Live) のインストール

Homebrewを使って、MacTeX（実体はTeX Live）をインストールします。

1. 以下のコマンドを実行します。
    
    ```bash
    brew install --cask mactex
    ```
    
    - パスワードを求められたら、Macのログインパスワードを入力します（画面には表示されませんが入力されています）。

> ⚠️ 注意：時間と容量について
MacTeXは非常に巨大（約8〜10GB）です。
TeX Liveの膨大なライブラリをすべてダウンロードするため、回線速度にもよりますが30分〜1時間程度かかります。
焦らず、この間に他の課題を済ませるなどして気長に待ちましょう。
> 

### Step 3: 動作確認と初期設定

インストール完了後、ターミナルを再起動（一度閉じて開き直す）し、正しく「TeX Live」が認識されているか確認します。

1. **バージョン確認**
    
    ```bash
    tex -v
    ```
    
    - `TeX 3.141592653 (TeX Live 2025)` のように表示されれば成功です。
2. **用紙サイズの固定（A4）**
デフォルトが米国規格（Letter）になっている場合があるため、日本の標準であるA4に設定します。
    
    ```bash
    sudo tlmgr paper a4
    ```
    
3. **パッケージのアップデート**
インストール直後でも更新がある場合があります。最新の状態にしておきましょう。
    
    ```bash
    sudo tlmgr update --self --all
    ```
    

> 📝 補足：TeX Liveの年度管理について
TeX Liveは年度単位で管理されています（TeX Live 2025など）。年度を跨いだ大規模更新に失敗した場合は、アップデートではなく「再インストール」が推奨されます。
> 

これで、あなたのMacに本格的な組版環境が整いました。
次は、この環境を快適に操作するためのエディタを導入します。

> ⚠️ うまくいかない場合は
インストールでエラーが出た場合は、第6章「トラブルシューティング」を参照してください。

# 第3章 エディタ導入：VS Code または Cursor

2025年現在、LaTeXの執筆環境として推奨されるのは、世界標準の **Visual Studio Code (VS Code)**、あるいはそれをベースにAI機能を強化した **Cursor** です。

これらは単なるメモ帳ではありません。強力な拡張機能によって、LaTeXの執筆、コンパイル、プレビューまでを一つの画面で完結させる「統合開発環境（IDE）」となります。

### 1. エディタの選択とインストール

どちらを選んでも、後述する設定や使い勝手はほぼ同じです。好みで選んでください。

> 📝 補足：数学科環境での推奨について
数学科の公式環境としては VS Code を推奨します。Cursorは個人利用の範囲で使用してください。
> 
- **パターンA: Visual Studio Code (VS Code)【推奨】**
    - **特徴:** Microsoft製。圧倒的なシェアと安定性を誇る業界標準。
    - **インストール:** `brew install --cask visual-studio-code`
- **パターンB: Cursor（カーソル）**
    - **特徴:** AI特化型エディタ。GPT-4などのAIによるコード修正が得意。VS Codeの拡張機能がそのまま使えます。
    - **インストール:** `brew install --cask cursor`

> Technical Note: Cursorユーザーへの注意点
CursorはVS Codeベースですが、一部挙動が異なります。
- コマンド: 初回起動時に code コマンド（ターミナルから開く機能）が cursor コマンドとして登録される場合があります。
- UI: 拡張機能はVS Codeと互換性がありますが、設定画面のUIなどが独自のものになっている場合があります。「基本はVS Codeと同じ」ですが、画面が少し違うことに戸惑わないでください。
> 

### 2. 日本語化（Japanese Language Pack）

デフォルトではメニューが英語ですので、日本語化します。

1. エディタを起動し、左端のブロック崩しのようなアイコン（**Extensions / 拡張機能**）をクリック（ショートカット: `[Cmd]` + `[Shift]` + `[X]`）。
2. 検索窓に `Japanese` と入力。
3. **「Japanese Language Pack for Visual Studio Code」** を選択し、Installをクリックして再起動します。

### 3. 拡張機能「LaTeX Workshop」の導入

これこそが、汎用エディタを「LaTeX専用機」に変える強力なプラグインです。

- **構文ハイライト:** コマンドを色分けして見やすくします。
- **自動コンパイル:** ファイルを保存するだけで、バックグラウンドでビルドツールを走らせ、PDFを作成します。
- **PDFプレビュー:** エディタ内でPDFを表示し、クリックした場所のコードへジャンプ（SyncTeX）できます。

**導入手順:**
拡張機能タブで `LaTeX Workshop` を検索し、**Install** します。

> ⚠️ うまくいかない場合は
拡張機能のインストールや設定でエラーが出た場合は、第6章「トラブルシューティング」を参照してください。

# 第4章 実践：マルチエンジン対応の執筆環境構築

エディタの準備ができたら、いよいよ**本格的な執筆環境**を一気に構築します。

ここでは、**LuaLaTeX**（モダン標準）、**upLaTeX**（学術標準）に加え、古い**pLaTeX**や海外で使われる**XeLaTeX**も含めた、あらゆるTeX環境に対応できる「完全なマルチエンジン環境」を構築します。

### Step 1: コンパイル設定ファイル `.latexmkrc` の作成

まず、ビルドツール `latexmk` に対する詳細な「指示書」を作成します。これはMacTeX（TeX Live）側に「どうやって変換するか」を教えるためのファイルです。

1. エディタ（VS Code / Cursor）で新規ファイルを作成します（`[Cmd]` + `[N]`）。
2. 以下のコードをコピー＆ペーストします。

**ファイル名: .latexmkrc**

```perl
#!/usr/bin/perl

# 1. 各エンジンのコマンド定義
# -synctex=1             : PDFとエディタの相互ジャンプ機能(SyncTeX)を有効化
# -interaction=nonstopmode : エラーが発生してもコンパイルを中断せずに続行（自動ビルドを止めないため）
# -file-line-error       : エラーメッセージに「ファイル名:行番号」を表示し、デバッグしやすくする

$latex            = 'uplatex -synctex=1 -interaction=nonstopmode -file-line-error %O %S';
$lualatex         = 'lualatex -synctex=1 -interaction=nonstopmode -file-line-error %O %S';
$xelatex          = 'xelatex -synctex=1 -interaction=nonstopmode -file-line-error %O %S';

# 2. 補助ツールの定義
$bibtex           = 'upbibtex %O %B';       # 参考文献処理。日本語対応版のupbibtexを指定
$biber            = 'biber %O --bblencoding=utf8 -u -U --output_safechars %B'; # 新しいBibLaTeX用ツール
$makeindex        = 'upmendex %O -o %D %S'; # 索引作成。辞書順ソート精度の高いupmendexを指定

# 3. PDF変換ドライバの定義 (DVI -> PDF)
# uplatexが出力したDVIファイルをPDFに変換するドライバ
$dvipdf           = 'dvipdfmx %O -o %D %S';
```

1. **保存:** `[Cmd]` + `[S]` を押し、**ホームディレクトリ**（`/Users/あなたの名前/`）に `.latexmkrc` という名前で保存します。
    - **重要:** ファイル名の先頭のドット `.` を忘れずに。保存時に「“.”から始まる名前を使用しますか？」と聞かれたら「“.”を使用」を選んでください。

### 【技術解説】このコードは何をしているのか？

これは「おまじない」ではありません。各行には明確な技術的意図があります。

- **`$latex = 'uplatex ...'`**
    - 従来のDVI形式を出力する際のデフォルトエンジンとして、Unicode完全対応の **`uplatex`** を指定しています。
    - オプションの `synctex=1` は、PDFをクリックしてソースコードへ飛ぶ機能のために必須です。
    - `interaction=nonstopmode` は、エラーが出るたびに「?」と聞いて止まってしまうのを防ぎます。
- **`$lualatex = 'lualatex ...'`**
    - LuaLaTeXエンジンにも同様のオプションを適用しています。こちらはDVIを経由せず、直接PDFを生成します。
- **`$bibtex = 'upbibtex ...'`**
    - 参考文献リストを作成するコマンドです。古い `bibtex` は日本語を扱えないため、多言語対応の **`upbibtex`** を指定します。
- **`$biber = 'biber ...'`**
    - 現代的なBibLaTeXを使うためのツールです。Unicodeをネイティブに扱えます。
- **`$makeindex = 'upmendex ...'`**
    - 索引（Index）を自動生成するコマンドです。従来の `mendex` よりも、Unicode文字の「あいうえお順」ソート精度が高い **`upmendex`** を採用しています。
- **`$dvipdf = 'dvipdfmx ...'`**
    - `uplatex` が出力したDVIファイルを、PDFに変換する「ドライバ」を指定しています。画像が表示されないトラブルの多くは、この工程の不備が原因です。

> Technical Note: 設定ファイルの読み込み順序と優先度
複数のプロジェクトを抱えたとき、「基本はLuaLaTeXだが、この卒論フォルダだけはupLaTeXにしたい」という場面があります。latexmk の設定ファイル読み込みルールを知っていれば、これをスマートに解決できます。
> 
> 1. **ホームディレクトリ (`~/.latexmkrc`):**
> 今回作成したファイル。Mac全体の「デフォルト設定」になります。
> 2. **カレントディレクトリ (`./.latexmkrc`):**
> 作業中の `.tex` ファイルと同じフォルダにある設定ファイル。**こちらが優先されます。**
> 
> **活用法:**
> 特殊な設定が必要なプロジェクトフォルダの中にだけ、別の `.latexmkrc` を作成して設定を書き込めば、そのフォルダ内でのみ設定が上書き（オーバーライド）されます。
>

### Step 2: エディタ設定 (`settings.json`) の記述

次に、エディタ側に「どのエンジンを使うか」を選べるレシピを登録します。
ここでは、**pLaTeX** や **XeLaTeX** も含めた包括的な設定を行います。

**重要なルール:**`"latex-workshop.latex.recipes"` のリストの**一番上に書かれたレシピが「デフォルト」**として採用されます。普段使いのエンジンを先頭に配置してください。

1. 設定画面 (`[Cmd]` + `,`) 右上の**「設定(JSON)を開く」**アイコンをクリック。
2. 波括弧 `{ }` の中に、以下のコードを追記してください。

- **パターンA: デフォルト設定（settings.jsonの順序）**

実際にファイルを作成し、マルチエンジン環境が正しく動くか確認しましょう。

> Technical Note: 文書クラスの選び方
初心者がやりがちなミスの一つに、「どんなエンジンでもとりあえず jsarticle を使う」というものがあります。文書クラス（\documentclass）は、使用するエンジンと相性があります。
> 
> 
> 
> | 文書クラス | 対応エンジン | 特徴 | 推奨用途 |
> | --- | --- | --- | --- |
> | **`jlreq`** | upLaTeX / LuaLaTeX / pLaTeX | **エンジン非依存**。カスタマイズが容易。W3C「日本語組版の要件」準拠。 | **自由形式のレポート（本ガイド推奨）** |
> | `jsarticle` / `jsbook` | pLaTeX / upLaTeX | 伝統的で情報が豊富。学会テンプレートの多くがこれベース。 | 学会・大学指定テンプレートがある場合 |
> | `ltjsarticle` / `ltjsbook` | LuaLaTeX専用 | jsclassesのLuaLaTeX版。 | LuaLaTeXで既存jsclasses資産を使いたい場合 |
> 
> **結論：**
> - 学会・大学指定のテンプレートがある → **テンプレートに従う**（多くはjsclasses）
> - 自由に書ける → **jlreqを推奨**（エンジンを変えてもコードを変えなくて済む）
> - ネットのサンプルコードを参考にしたい → jsclassesでも可（情報量が多い）
> 

> ⚠️ うまくいかない場合は
コンパイルでエラーが出た場合は、第6章「トラブルシューティング」を参照してください。特に「日本語が文字化けする」「PDFが生成されない」などの症状別に対処法をまとめています。

# 第5章 理論：TeXエンジンの進化と日本語処理の仕組み

> この章について
本章は、TeXの内部構造を深く理解したい方向けの発展的な内容です。環境構築を急ぐ方は、この章を飛ばして第6章へ進んでも問題ありません。 トラブルに遭遇したときや、「なぜこの設定が必要なのか」を知りたくなったときに戻ってきてください。
> 

前章までで環境構築は完了しましたが、数理系学生として一歩進んだ理解を得るために、TeXエンジンの「系譜」と「日本語処理の仕組み」について、その内部構造から深く学びます。

これを知っておくと、不可解なエラーや文字化けに遭遇した際、あるいは特殊なフォントやパッケージを使いたい際に、その挙動を論理的に推測できるようになります。**次章の「ルール」がなぜ必要なのかも、この章を読めば納得できるでしょう。**

## 1. 基礎概念：エンジンとフォーマットの分離

まず、初学者が最初に混乱する「TeX」という言葉の定義を明確にします。

- **TeX（エンジン）:**
Donald E. Knuth教授が開発した、組版処理を行う実行プログラムそのものです。テキストを読み込み、箱（Box）とグルー（Glue）の概念を用いて文字配置を計算します。
- **LaTeX（フォーマット）:**
Leslie Lamport氏が開発した、TeXの上で動くマクロパッケージです。TeX本来の命令（プリミティブ）は非常に低レベルで扱いづらいため、人間が使いやすいコマンド（`\section`, `\documentclass` など）として体系化したものです。

私たちが「TeXを書く」と言うとき、技術的には**「LaTeXフォーマットのマクロを記述し、特定のTeX派生エンジンで処理させる」**ことを指します。

## 2. 日本語化の夜明け：pTeX (Publishing TeX)

オリジナルのTeXは欧文（7bit/8bitコード）しか想定していませんでした。これを日本語に対応させるため、アスキー社などが中心となって開発したのが **pTeX** です。

単に日本語が表示できるだけでなく、処理系自体を改造し、以下のような機能を実装しました。

1. **縦書き組版**: 世界でも珍しい「縦書き」の座標計算と文字配置を実装。
2. **日本語禁則処理**: 行頭の「。」や行末の「（」などを禁止する処理。
3. **和欧混植処理**: 日本語（全角）と欧文（半角）が隣り合った際、文字間に適切な空白（四分アキなど）を自動挿入する機能（JFM: Japanese Font Metricによる制御）。

理系の文書では数式（欧文扱い）と日本語が頻繁に混在するため、pTeXによるこのサポートは不可欠でした。

## 3. 機能拡張の流れ：e-TeX と e-pTeX

### e-TeX (Extended TeX)

一方で、pTeXとは別の文脈で、本家のTeX自体を拡張しようという動きがありました。それが **e-TeX** です。
TeXは古い言語であるため、変数を格納する「レジスタ」の数に厳しい制限（256個など）がありましたが、e-TeXではこれを大幅に増加させ、新たな条件式なども追加しました。

### e-pTeX (e-TeX + pTeX)

「pTeX（日本語機能）」と「e-TeX（拡張機能）」という2つの進化を合体（マージ）させたのが **e-pTeX** です。
現在、ターミナルで `platex` コマンドを打つと、以下のようなバージョン情報が表示されます。

```bash
This is e-pTeX, Version 3.1415926-p3.4-110825-2.6 (utf8.euc) ...
restricted \write18 enabled.
```

## 4. Unicodeの壁と upTeX の革新

### e-pTeX の限界：文字コード問題

e-pTeXは優秀でしたが、内部処理が「JIS/EUC」などの古い文字コードベースでした。近年標準となった **Unicode (UTF-8)** に完全対応しておらず、以下のような問題がありました。

1. **入力変換の必要性**: かつては `nkf` などのツールを用い、TeXファイルをEUCコード等に変換してから処理系に渡す必要がありました。
2. **「髙」の消滅**: JIS第一・第二水準外の文字（「髙（はしごたか）」や中国語など）は、内部コードにマッピングできず、出力時に**亜空間へと消滅**します。

これを回避するために `otf` パッケージを使い、`\UTF{9AD9}` のようにコード値を直接入力する方法もありましたが、直感的ではありません。

### upTeX / e-upTeX (Unicode pTeX)

そこで、pTeXをさらに改造し、**「内部文字コードをUnicode化」**したのが **upTeX** です。さらにe-TeXとマージしたものが **e-upTeX** となります。
これにより、ソースコードにそのまま「髙」と書いても正しく出力されるようになりました。

現代において、pTeX系のエンジンを使うならば、互換性とUnicode対応を両立した **e-upTeX (upLaTeX)** が最適解です。

## 5. 出力の仕組み：DVIファイルとドライバ

pTeX/upTeX系を理解する上で最も重要なのが、**「DVIファイル」と「ドライバ」**による2段階処理です。

LaTeXは「テキストを入れるとPDFが出てくる魔法の箱」ではありません。**複数のプログラムが連携して動く「工場（パイプライン）」**です。この仕組みを知らないと、「画像が出ない」「文字化けする」といったトラブルに自力で対処できません。

### 処理パイプラインの全体像

LaTeXには大きく分けて、**「DVI（中間ファイル）を経由するルート」**と**「PDFを直接出力するルート」**の2種類があります。

**Type A: DVIを経由するルート（uplatex, platex）**

```
[.tex] → [uplatex (e-upTeX)] → [.dvi] → [dvipdfmx] → [.pdf]
                                            ↑
                                      画像・フォント
```

1. **Engine (e-upTeX):** ソースコードを読み、文字の配置情報だけが書かれた DVI ファイルを出力します。この時点では**画像の実体やフォントデータは含まれていません**。
2. **Driver (dvipdfmx):** DVI を読み込み、ここで初めて画像ファイルやフォントを結合して PDF を作成します。

**Type B: PDFを直接出力するルート（lualatex, pdflatex）**

```
[.tex] → [lualatex (LuaHBTeX)] → [.pdf]
              ↑
        画像・フォント
```

- 中間ファイル（DVI）を作らず、エンジンが直接 PDF を書き出します。
- ドライバの指定忘れによるトラブルが起きません。

### DVI (DeVice Independent) とは

エンジン（e-upTeXなど）は、文字の**「位置情報（座標）」**だけを計算し、それを記録した中間ファイル **`.dvi`** を出力します。この時点では、フォントの実体や画像データは埋め込まれていません。

### ドライバの役割

このDVIファイルを、実際のプリンタやPDF形式に合わせて変換するのが**ドライバ**の役割です。日本では **dvipdfmx** が標準的に使われます。

1. **コンパイル**: `uplatex` が `.tex` を読み、`.dvi` を出力。
2. **変換**: `dvipdfmx` が `.dvi` を読み、フォントファイルや画像を結合して `.pdf` を出力。

> Technical Note: e-pTeXなどでフォント埋め込みを行う際、TeX側の設定（otf パッケージなど）と、ドライバ側の設定（kanji-config-updmap など）の整合性が取れていないとエラーになります。これが「TeXは環境構築が難しい」と言われる一因です。
> 

## 6. 海外発のモダンなエンジン

### XeTeX（ズィーテフ）

- **OSフォントの直接利用**: デバイス（PC）にあるフォントを直接扱えるようにしました。
- **XDV**: DVIを拡張した **XDV** 形式を経由してPDF化します。
- **評価**: 日本語表示には `xeCJK` パッケージを使いますが、pTeX系ほど厳密な日本語組版が得られない場合があります。

### LuaTeX（ルアテフ）

pdfTeXの後継であり、将来の標準と目されています。

- **特徴**: エンジン内部にプログラミング言語 Lua を組み込んでおり、TeXの挙動をスクリプトで制御できます。
- **評価**: **LuaTeX-ja** プロジェクトにより、pTeXのような「エンジンの改造」ではなく、Luaスクリプトによる「機能追加」として日本語組版を実現しています。「直接PDF出力」によるトラブルの少なさと、pTeX譲りの組版品質を両立しており、特にTikZなどの作図ツールとの親和性が高いです。

## 7. まとめ：数理系学生の選択指針

| エンジン | 内部処理 | 出力形式 | 特徴 | 推奨用途 |
| --- | --- | --- | --- | --- |
| **e-upTeX** (upLaTeX) | Unicode | DVI経由 | **日本の学術標準**。過去の資産（jsclasses）と完全互換。「髙」もOK。**コンパイルが高速。** | 大学指定のレポート、学会論文。**本ガイドの推奨。** |
| **LuaTeX** (LuaLaTeX) | Unicode | 直接PDF | **モダン標準**。Luaによる拡張性と直接出力の安定性。TikZと好相性。ただし**コンパイルがやや遅い**。 | 自由形式のレポート、スライド、自作文書。 |
| **XeTeX** | Unicode | XDV経由 | **フォント重視**。OSのフォントが簡単に使える。日本語組版はやや甘い。 | 特殊なフォントを使いたいデザイン重視の文書。 |

> 本ガイドでupLaTeXをデフォルトにしている理由
1. コンパイル速度: LuaLaTeXは初回コンパイルに時間がかかることがあります
2. 互換性: 学会や大学指定のテンプレートはupLaTeX/pLaTeX前提のものが多い
3. 安定性: 長年使われてきた実績があり、情報も豊富
> 
> 
> LuaLaTeXは将来性のあるモダンなエンジンですが、まずはupLaTeXで基礎を固め、必要に応じてLuaLaTeXに移行するのが堅実な道です。
>

# 第6章 ルール：LaTeXの「新常識」とトラブルシューティング

ここでは、初心者が陥りやすい「エラーの罠」と、プロが見れば一発でバレる「素人っぽい組版」を回避するためのルールを紹介します。

**前章の理論を踏まえると、これらのルールがなぜ必要なのかが理解できるはずです。**

### 1. 現代LaTeXの「新常識」（エラー回避編）

### ルール1: おまじない `plautopatch` を必ず入れる【必須】

pLaTeX/upLaTeX は、海外製の便利なパッケージ（hyperref など）と相性が悪いことがあり、そのまま使うとエラーになります。これを自動修復するパッチを読み込みましょう。

**なぜ必要か？** 第5章で説明した通り、pTeX系は日本独自の拡張です。海外で開発されたパッケージは pTeX の存在を想定していないため、互換性の問題が生じます。`plautopatch` はこの差分を自動的に埋めてくれます。

- **対応:** ファイルの**一番最初（\documentclass より前）**に以下を書く。
    
    ```latex
    \RequirePackage{plautopatch}
    \documentclass[uplatex,dvipdfmx]{jlreq}
    ```
    

### ルール2: 「ドライバ」を明示する【必須】

「画像が出ない」「図の位置がおかしい」。このトラブルの主犯は、DVIドライバ（`dvipdfmx`）の指定忘れです。

**なぜ必要か？** 第5章で説明した「DVI → PDF」の2段階処理において、どのドライバで変換するかをTeXに伝える必要があります。これを省略すると、TeXは古いドライバ（dvips等）を想定してしまい、画像パスの解釈などがずれてしまいます。

- **対応:** `\documentclass` のオプションに必ず `dvipdfmx` を書く。
    
    ```latex
    % 良い例：クラスオプションで一括指定（すべてのパッケージに伝わる）
    \documentclass[uplatex,dvipdfmx]{jlreq}
    \usepackage{graphicx}
    \usepackage{color}
    ```
    

> ⚠️ インターネット上の古い記事について
インターネット上には、「\usepackage[dvips]{graphicx}」のように dvips などの異なるドライバを指定している古い記事が散見されます。dvipdfmx を使う環境でこれを真似すると、ドライバが競合してエラーになります。
「ドライバ指定は \documentclass に dvipdfmx と書く、それ以外では書かない」 と統一するのが最も安全です。
> 

### ルール3: ファイル名・フォルダ名に「スペース」と「日本語」は使わない【必須】

意外と見落としがちなエラーの原因です。TeXのツール群は、ファイルパスにスペースや日本語が含まれているとうまく動作しないことがあります。

- **× Bad:** `レポート 最終版.tex` （スペース、日本語あり）
- **○ Good:** `report_final.tex` （半角英数とアンダースコアのみ）

### ルール4: 画像形式は「PDF, PNG, JPEG」の3択【必須】

かつては「ベクター画像ならEPS」が常識でしたが、現在は**「ベクター画像ならPDF」**が標準です。

| 用途 | 推奨形式 | 備考 |
| --- | --- | --- |
| 図形・グラフ・イラスト | **PDF** | 拡大しても劣化しない |
| スクリーンショット | **PNG** | 背景透過が可能 |
| 写真 | **JPEG** | 複雑な色味に適する |

**画像の取り込み方:**

```latex
\usepackage{graphicx}

% 拡張子は書かない（同名のPDFが優先的に読まれる）
\includegraphics[width=8cm]{figure}
```

**なぜEPSを避けるべきか？**
EPSを使うにはGhostscriptのインストールやパス設定が必要で、環境依存のトラブルが起きやすいです。手元にEPSがある場合は、Macの「プレビュー.app」で開いて「ファイル」→「PDFとして書き出す」で変換してから使うのが最も確実です。

### 2. トラブルシューティング：困ったときは

エラーが出ても焦る必要はありません。エンジニアのようにログを読み解きましょう。

### 症状1: ビルドが止まる・終わらない

- **原因:** コマンド入力ミスで、TeXがユーザーからの入力を待っている状態（`?` プロンプト）かもしれません。
- **対処:** ターミナルなら `x` を押してEnter。VS Codeなら、設定で `interaction=nonstopmode` が入っているか確認してください（本ガイドの設定なら入っています）。

### 症状2: 「Undefined control sequence」エラー

- **原因:** コマンドのスペルミス（例: `\section` を `\sction` と書いた）か、必要なパッケージ（`\usepackage{...}`）の読み込み忘れです。
- **対処:** エラーメッセージに「行番号」が出ているはずです。その行をよく確認しましょう。

### 症状3: 全角スペースの罠

- **原因:** 見えない敵、**全角スペース**（　）がコードに含まれています。TeXはこれを「和文文字」として扱おうとし、コマンドの途中にあるとエラーになります。
- **対処:** VS Codeの設定で「全角スペースをハイライトする」拡張機能を入れるか、設定で可視化しましょう。

### 症状4: 日本語が文字化けする

- **原因:** ファイルの文字コードがUTF-8になっていない、またはエンジンと文書クラスの組み合わせが不適切です。
- **対処:**
    1. VS Codeの右下に表示されている文字コードを確認し、「UTF-8」になっていなければクリックして変換
    2. 第4章の「文書クラスの選び方」を参照し、エンジンに合ったクラスを使用しているか確認

### 症状5: 数式が表示されない・崩れる

- **原因:** `amsmath` パッケージの読み込み忘れ、または数式環境の記法ミスです。
- **対処:**
    1. プリアンブルに `\usepackage{amsmath}` があるか確認
    2. `\[...\]` や `\begin{equation}...\end{equation}` の閉じ忘れがないか確認

### 症状6: 図表番号や参照が「??」になる

- **原因:** これはバグではなく**仕様**です。LaTeXは複数回のコンパイルが必要な場合があります。
- **対処:** もう一度コンパイルを実行してください。本ガイドの設定（latexmk）を使っていれば自動的に解決されます。

> Technical Note: なぜ複数回コンパイルが必要なのか？
> 
> 
> LaTeXはソースコードを**上から下へ1回しか読みません**。
> 
> 1. **1回目のコンパイル:**
>     - 本文中の `\label{eq:1}`（数式番号の定義）を見つけると、「eq:1 は 式番号(1) だな」とメモ帳（`.aux` ファイル）に書き込みます。
>     - この時点では、前の方に出てきた `\ref{eq:1}`（参照）はまだ答えを知らないので `??` と表示します。
> 2. **2回目のコンパイル:**
>     - 最初に `.aux` ファイル（前回のメモ）を読み込みます。
>     - 「eq:1 は (1) だ」と知った状態で本文を処理するため、`\ref{eq:1}` が正しく (1) に置き換わります。
> 
> **latexmk** は、`.aux` ファイルなどの変化を監視し、「結果が変わらなくなるまで（収束するまで）」自動でコンパイルを繰り返してくれるツールです。本ガイドの設定を使っていれば、この問題を意識する必要はありません。
> 

### 3. 数学の常識・非常識（組版の美学編）

ここからは「コンパイルは通るが、読み手にとって美しくない」文書を避けるための話です。

東北大学の小田忠雄先生が記されたガイドラインにもある通り、ただ数式を表示するだけでは不十分です。**「意味の通る美しい数式」**を書くためのルールがあります。

### ルール5: `$$ ... $$` を使わない【禁止】

LaTeXでディスプレイ数式（別行立て数式）を書くとき、`$$ ... $$` は**非推奨**です。

```latex
% × 悪い例（TeXの古い記法）
$$
  E = mc^2
$$

% ○ 良い例（LaTeXの正式な記法）
\[
  E = mc^2
\]

% ○ 番号付きの場合
\begin{equation}
  E = mc^2
\end{equation}
```

`$$ ... $$` はPlain TeXの記法であり、LaTeXでは行間が不揃いになったり、他のパッケージと干渉する原因になります。

### ルール6: `\\` の乱用を避ける【禁止】

改行コマンド `\\` を行頭で連発して縦方向のスペースを作るのは避けましょう。

```latex
% × 悪い例
\section{はじめに}
\\
\\
本論文では...

% ○ 良い例
\section{はじめに}
\vspace{1em}  % 明示的に縦スペースを指定
本論文では...
```

### ルール7: 手動スペース `\,` `\!` の乱発を避ける【禁止】

数式内のスペース調整を `\,` や `\!` で手動で行うのは、最終手段です。

```latex
% × 悪い例（手動で微調整）
$\int\!\!\!\int f(x,y)\, dx\, dy$

% ○ 良い例（適切なコマンドを使う）
$\iint f(x,y)\, dx\, dy$  % amsmathの\iintを使う
```

まずは適切な環境やコマンドがないか調べ、それでも対応できない場合にのみ手動調整を行いましょう。

### ルール8: 「省略のピリオド」と「文末のピリオド」を区別する【推奨】

TeXは、ピリオド `.` の後ろに「文が終わった」と判断して広めのスペース（空白）を入れます。しかし、`Mr.` や `Fig.` の後ろに広い空白が入ると不格好です。

| **入力コード** | **出力結果のイメージ** | **判定** |
| --- | --- | --- |
| `Mr. Smith` | Mr.　Smith | **×** (空白が広すぎる) |
| `Mr.~Smith` | Mr. Smith | **○** (適切な空白) |
| `J. Math. Soc. Japan` | J.　Math.　Soc.　Japan | **×** (ガタガタ) |
| `J.~Math.~Soc.~Japan` | J. Math. Soc. Japan | **○** (美しい) |

### ルール9: 数式中の「普通の単語」は斜体にしない【推奨】

数式モード `$ ... $` の中は、変数を表すために自動的に**イタリック体（斜体）**になります。しかし、「if」や「const」のような単語まで斜体にしてはいけません。

- **× Bad:** `$if x > 0$` → *if* と *x* という2つの変数の掛け算に見えてしまう
- **○ Good:** `$\text{if } x > 0$` → ローマン体で正しく読める

### ルール10: 関数名はコマンドを使う【推奨】

高校数学までは気にならなかったかもしれませんが、`sin` や `log` を単に文字として打つのはNGです。

- **× Bad:** `$sin x$` → *s*, *i*, *n* の積に見える
- **○ Good:** `$\sin x$` → 適切なフォントと空白が入る

# 第7章 技術：論文執筆を加速するプロの技（参考文献・分割管理・Beamer）

卒論や修論のような大規模な文書を書く際、すべての文章を1つのファイルに書くのは非効率です。また、参考文献を手動で管理するのはミスの元です。
ここでは、研究者が実践している効率的なテクニックを紹介します。

### 1. 参考文献の自動管理（BibTeX / BibLaTeX）

参考文献リストを自動生成する仕組みです。第4章の設定（`.latexmkrc`）で両方とも使えるようにしてあります。

### BibTeX vs Biber：どちらを選ぶべきか？

| 項目 | BibTeX (upBibTeX) | Biber (BibLaTeX) |
| --- | --- | --- |
| **歴史** | 古くからある標準 | 最新の仕組み |
| **Unicode対応** | △（日本語ソートが苦手） | ◎（完全対応） |
| **URL/DOI処理** | △（手動設定が必要） | ◎（簡単に出力） |
| **カスタマイズ性** | △ | ◎ |
| **学会テンプレート** | ◎（多くが対応） | △（まだ少ない） |
| **推奨用途** | 学会・大学指定テンプレート | 自由形式のレポート |

**結論：**
- 学会・大学指定のテンプレートがある → **BibTeX (upBibTeX)** に従う
- 自由に書ける → **BibLaTeX + Biber** を推奨（Unicode完全対応で将来性がある）
- 迷ったら → **BibTeX (upBibTeX)**（互換性が高くトラブルが少ない）

### 共通：文献データベース（.bibファイル）の作成

どちらの方法でも、まず `myref.bib` というファイルを作り、文献情報を記述します。

```
@book{knuth84,
  author    = {Donald E. Knuth},
  title     = {The TeXbook},
  year      = {1984},
  publisher = {Addison-Wesley}
}

@book{okumura2023,
  author    = {奥村 晴彦 and 黒木 裕介},
  title     = {LaTeX2ε美文書作成入門},
  edition   = {改訂第9版},
  year      = {2023},
  publisher = {技術評論社}
}
```

### パターンA: BibTeX（伝統的な方法）

学会指定のテンプレートなどで使われます。upLaTeX/pLaTeXで一般的です。

1. **プリアンブルでの設定:** 特になし（標準で使えます）
2. **本文での引用:** `\cite{knuth84}` と書けば、自動的に番号が振られます。
3. **文末での出力:**
    
    ```latex
    \bibliographystyle{junsrt} % スタイル指定（junsrt: 出現順、jplain: 著者名順）
    \bibliography{myref}       % .bibファイルの読み込み（拡張子なし）
    ```
    

### パターンB: BibLaTeX + Biber（モダンな方法）

Unicodeに強く、カスタマイズが容易な現代的な方法です。upLaTeXでもLuaLaTeXでも使えます。

1. **プリアンブルでの設定:**
    
    ```latex
    \usepackage[style=numeric, backend=biber]{biblatex}
    % style=numeric: 番号形式 [1], [2]...
    % style=authoryear: 著者年形式 (Knuth 1984)
    \addbibresource{myref.bib} % 拡張子 .bib まで書く
    ```
    
2. **本文での引用:**
    
    ```latex
    \cite{knuth84}           % → [1] または Knuth 1984（styleによる）
    \parencite{knuth84}      % → (Knuth 1984) ※authoryearスタイル時
    \textcite{knuth84}       % → Knuth (1984) は〜と述べている
    ```
    
3. **文末での出力:**
    
    ```latex
    \printbibliography
    ```
    

### 2. 大規模文書の分割管理

数百ページの論文を1つの `main.tex` に書くと、スクロールが大変で、コンパイルも遅くなります。章ごとにファイルを分けるのが定石です。

### ファイル構成の例

```
thesis/
  ├── main.tex       (親ファイル)
  ├── chapters/
  │   ├── intro.tex  (第1章)
  │   └── theory.tex (第2章)
  └── ...
```

### 設定方法

1. **親ファイル (`main.tex`)** に全体の構造を書きます。
    
    **upLaTeXの場合（本ガイド推奨）:**
    
    ```latex
    \RequirePackage{plautopatch}
    \documentclass[uplatex,dvipdfmx,book]{jlreq}
    \begin{document}
      \input{chapters/intro.tex}
      \input{chapters/theory.tex}
    \end{document}
    ```
    
    **LuaLaTeXの場合:**
    
    ```latex
    \documentclass[book]{jlreq}
    \begin{document}
      \input{chapters/intro.tex}   % 拡張子は省略可能
      \input{chapters/theory.tex}
    \end{document}
    ```
    
    ※ jlreqはエンジン非依存なので、`[book]`オプションを付ければ書籍形式になります。
    
2. **子ファイル (`intro.tex`)** には中身だけを書きます（`\documentclass`などは不要）。
    
    ```latex
    \chapter{はじめに}
    ここに本文を書きます...
    ```
    

### Technical Note: 分割時のオートコンパイル

ファイルを分割すると、「子ファイル (`intro.tex`) を編集中に保存しても、親ファイル (`main.tex`) がコンパイルされない」という問題が起きがちです。

これを解決するには、**マジックコメント**を子ファイルの先頭に入れます。

**`intro.tex` の先頭:**

```latex
%!TEX root = ../main.tex
\chapter{はじめに}
...
```

これにより、VS Codeは「このファイルの親玉は `main.tex` だな」と認識し、子ファイルを保存した瞬間に親ファイル全体をコンパイルしてくれます。

### 3. 発表スライドの作成（Beamer）

学会発表やゼミ発表で使うスライドもLaTeXで作成できます。**Beamer**は、LaTeXでプレゼンテーションスライドを作成するための標準的な文書クラスです。

### Beamerのメリット

- 数式が美しく表示される（PowerPointでの数式入力の苦労から解放）
- 論文と同じソースコードから図表を流用できる
- バージョン管理（Git）との相性が良い
- テーマを変えるだけでデザインを一括変更できる

### 基本的な構造

```latex
%!TEX program = uplatex
\RequirePackage{plautopatch}
\documentclass[uplatex,dvipdfmx]{beamer}

% テーマの設定
\usetheme{Madrid}           % スライドのデザイン
\usecolortheme{default}     % 配色

% 日本語フォントの設定
\usepackage{pxjahyper}      % PDFのしおりの文字化け防止

\title{ゼミ発表タイトル}
\author{山田 太郎}
\institute{◯◯大学 数学科}
\date{\today}

\begin{document}

\begin{frame}
  \titlepage
\end{frame}

\begin{frame}{目次}
  \tableofcontents
\end{frame}

\section{はじめに}

\begin{frame}{研究の背景}
  \begin{itemize}
    \item 項目1
    \item 項目2
  \end{itemize}
\end{frame}

\begin{frame}{主定理}
  \begin{theorem}
    任意の $\varepsilon > 0$ に対して...
  \end{theorem}

  \begin{proof}
    証明の概要...
  \end{proof}
\end{frame}

\end{document}
```

### LuaLaTeXで使う場合

```latex
%!TEX program = lualatex
\documentclass{beamer}
\usetheme{Madrid}
\usepackage{luatexja}       % 日本語サポート

\title{ゼミ発表タイトル}
\author{山田 太郎}
\date{\today}

\begin{document}
% ... 以下同様
\end{document}
```

### よく使うテーマ

| テーマ名 | 特徴 |
| --- | --- |
| `Madrid` | シンプルで見やすい。学会発表向き |
| `Berlin` | ヘッダーにナビゲーション表示 |
| `CambridgeUS` | 赤系の配色。落ち着いた印象 |
| `metropolis` | モダンでミニマル（別途インストール必要） |

> 📝 テーマ選びの参考サイト
- Beamer Theme Matrix: https://mpetroff.net/files/beamer-theme-matrix/
標準テーマ × カラーテーマの組み合わせを一覧できます。
- Beamer Theme Gallery: https://deic.uab.cat/~iblanes/beamer_gallery/
各テーマの実際の見た目を確認できます。
- Overleaf Gallery (Beamer): https://www.overleaf.com/gallery/tagged/beamer
大学・研究機関のテンプレートも多数あります。
> 

> 📝 補足：Beamerの注意点
- 1スライドに詰め込みすぎない（箇条書きは3〜5項目程度）
- アニメーション（\pause）は控えめに
- フォントサイズは大きめに（\Large等を活用）
- 学会によってはPowerPoint指定の場合もあるので、事前に確認を

| `fancyhdr` | ヘッダー・フッターのカスタマイズ | ページ番号位置の変更など |
| `setspace` | 行間の調整 | `\onehalfspacing`, `\doublespacing` |
| `enumitem` | 箇条書きのカスタマイズ | 項目間隔の調整など |

### その他便利なパッケージ

| パッケージ | 用途 | 備考 |
| --- | --- | --- |
| `physics` | 物理・数学の記法 | `\dv{f}{x}`, `\bra`, `\ket` |
| `siunitx` | SI単位の正しい表記 | `\SI{9.8}{m/s^2}` |
| `listings` | ソースコードの表示 | 構文ハイライト対応 |
| `algorithm2e` | アルゴリズムの擬似コード | 計算機科学系の論文向け |
| `xcolor` | 色の指定 | `\textcolor{red}{赤}` |

### パッケージ読み込みの注意点

1. **読み込み順序に注意**: 特に`hyperref`は最後（または最後付近）に読み込むのが原則です。
2. **重複機能の回避**: 似た機能のパッケージを複数読み込むと競合することがあります。
3. **jlreqとの相性**: jlreqクラスは多くの機能を内蔵しているため、不要なパッケージを読み込まないようにしましょう。
4. **CTANで確認**: 使い方がわからない場合は、[CTAN](https://ctan.org/)でパッケージ名を検索し、公式ドキュメントを参照してください。
